There is no impedance mismatch between Blueprints and OrientDB. However, there are a few peculiarities in OrientDB that make it not completely faithful to the "property graph model":http://github.com/tinkerpop/blueprints/wiki/Property-Graph-Model. These are itemized below.

# *No user defined element identifiers*: OrientDB is the gatekeeper and creator of vertex and edge identifiers. Thus, when creating a new vertex or edge, the provided object identifier is ignored.

The following is the list of @orientdb@ namespace functions that are provided with the Gremlin distribution.

# "OrientDB Functions":#orientdb_functions
  ** @graph orientdb:open(string)@

h2(#orientdb_functions). OrientDB Functions

h3. Open @graph orientdb:open(string, string?, string?)@

This function opens up an OrientDB graph in the provided string directory. If an OrientDB graph already exists at that directory, then that graph is loaded. Note that the last component of the path is the name of the graph, not the directory where data will be stored. The two optional string parameters are the username and password for the graph. If none are provided, "admin:admin" are assumed.

bc. gremlin> $_g := orientdb:open('/tmp/graph/test')
==>orientgraph[/tmp/graph/test]

While not an OrientDB function per se, its important to note that before quitting Gremlin, you should always shutdown your OrientDB graph instances to ensure that they are properly stored to disk. This rule of thumb goes for all non-memory based data management systems.
